## 8. Pattern Detection (Cross-Run)

### New Patterns Observed This Cycle

**PAT-007 (new) — F7 doc/spec-currency wave:** When multiple fix-bursts land in rapid
succession during F7 (this cycle: R1 doc fixes → R2 provenance sweep + #[non_exhaustive] →
R3 anchor drift → R4 BC Architecture Anchors + post-delivery notes), each burst generates
its own consuming-surface propagation gap, which the next adversary round finds. The
resulting cascade extends F7 by 1-2 rounds per burst. Root cause: fix-bursts during F7
are dispatched with a narrow remediation scope and do not include a post-burst consuming-
surface sweep. The DF-CONSISTENCY-AUDIT-POST-FIXBURST-001 policy exists but is not enforced
as a blocking checklist item at the point of dispatch.

**PAT-008 (new) — Release-config approval prompt self-falsification:** When the
`human_approval_prompt` in release-config.yaml embeds a point-in-time status snapshot
(e.g., "Round-2 pending / R1 findings documented / remediation-in-flight"), the prompt
self-falsifies by the time any subsequent round resolves the state it described. The opener
"All quality gates are satisfied" was copied from a prior template without matching the
held/in-progress status. Found in F7-R2 Pass C. Now codified: prompt must reference
STATE.md as source-of-truth, not embed inline status.

**PAT-009 (new) — Adversary stale git-ref cache:** Adversary subagents operating in
isolated sandboxes can report a stale git HEAD SHA while reviewing the correct live file
contents. This generates a false-alarm finding that consumes one remediation burst before
being closed. Occurred at least 3 times across this project (D-105, F7-E8-consistency,
F7-R4-Pass-C). Orchestrator supplying a verified HEAD SHA in the adversary dispatch
prompt eliminated the false alarm in R5 (adversary reviewed the same files, found 0
findings — no SHA discrepancy reported). This is the confirmed mitigation.

### Existing Pattern Updates

**PAT-001 (pr-manager shortstop):** Rate dropped to 1 occurrence (vs 5/5 in v0.7.0).
The relay-trust guardrail pattern held but the orchestrator adapted quickly by executing
the merge directly. Still OPEN for structural fix via STORY-121.

**PAT-002 (doc-tense recurrence):** Zero occurrences this cycle. May be genuinely
improving, or may be attributed to the cycle's implementation path (struct-reshape and
behavioral addition) not triggering the specific RED-phase doc-comment survival pattern
that generated v0.7.0's 7 recurrences.

**PAT-004 (consumer-sweep gap post-fixburst):** Direct recurrence — F7-R4 found BC
Architecture Anchors and story post-delivery notes not swept by PR #273. Policy
DF-CONSISTENCY-AUDIT-POST-FIXBURST-001 exists but was not enforced as a blocking checklist.
Total occurrences: 3 (v0.7.0 x2, v0.9.0 x1).

### PROP-E18-01 — Pre-F7 Doc/Spec-Currency Consolidated Sweep

**Category:** workflow
**Priority:** P1 HIGH
**Evidence:** F7 took 5 rounds; 0 code defects found; all 9 R1 findings + 4 R2 findings +
1 R3 finding + 2 R4 findings were documentation or spec-currency. R1+R2 alone (13 findings)
would have been eliminated by a single pre-F7 sweep. The CHANGELOG-bundle omission (PR
#270, partially; F7-R1 remainder) and release-config snapshot (F7-R2) are structurally
predictable for any multi-story bundled release.

**Recommendation:** Add a pre-F7 doc/spec-currency sweep step to the feature pipeline. Run
before the first F7 adversarial triple. Sweep checklist:

1. CHANGELOG [unreleased/next] — confirm all stories in the bundle are represented; remove
   stale forward-references from prior release entries.
2. release-config.yaml `human_approval_prompt` — confirm it references STATE.md as source
   of truth, not an embedded status snapshot. Confirm `required_checks` matches actual
   gating job names in ci.yml.
3. Help-text doc-comments (`/// ...` and `#[doc = "..."]` on clap fields) — confirm no
   BC-/STORY-/LESSON- factory provenance tags are present. (Now also gated by CI, but
   catching it at sweep time avoids a remediation PR.)
4. BC Architecture Anchors for any BC whose governed struct/enum was modified — confirm
   anchors describe shipped reality.
5. Story post-delivery notes — confirm any post-merge changes (e.g., `#[non_exhaustive]`
   additions, constructor changes) are recorded.

**Affected files:** `docs/orchestrator-feature-sequence.md` (or equivalent), plus any
pre-F7 template in the factory workflow.
**Risk:** Moderate (adds ~1 agent invocation per cycle; should reduce F7 rounds by 2-3).
**Estimated savings:** 2-3 F7 rounds per bundle-release cycle = ~10 agent invocations.

### PROP-E18-02 — Orchestrator Supplies Verified HEAD SHA in Adversary Dispatch

**Category:** agent (adversary dispatch)
**Priority:** P1 HIGH
**Evidence:** PAT-009 documented above. Stale git-ref cache in adversary subagent sandbox
produced a false-alarm SHA finding in F7-R4 Pass C (D-131). Orchestrator resolved it by
verifying `git rev-parse HEAD` directly and closing as false alarm. In F7-R5, the
orchestrator supplied the verified HEAD SHA in the dispatch prompt. R5 found 0 findings —
no SHA discrepancy reported. This is the confirmed mitigation. Same pattern occurred at
least twice prior (D-105, E-8 F7 consistency audit).

**Recommendation:** Modify adversary subagent dispatch templates to include the verified
HEAD SHA in the prompt preamble, sourced from `git rev-parse HEAD` at dispatch time. Format:

  "Verified develop HEAD at dispatch: `<SHA>`. If your git-ref tooling reports a different
   value, disregard the tooling output and treat `<SHA>` as ground truth for this pass."

This eliminates the false-alarm class entirely at zero adversarial quality cost.

**Affected files:** Adversary dispatch template or orchestrator dispatch step.
**Risk:** Low (purely additive; does not constrain adversary review scope).

### PROP-E18-03 — F4-Close Bundle CHANGELOG Audit Checklist Item

**Category:** workflow (pr-manager F4-close checklist)
**Priority:** P2 HIGH
**Evidence:** F5-R1 finding F-B-001 HIGH (CHANGELOG omitted STORY-122/A reshape and
STORY-119/B behavioral flip, falsely claimed byte-identical). F7-R1 finding cluster
(further CHANGELOG gaps). Root cause: when 2+ stories deliver under one unreleased version

**Recommendation:** At F4-close (all stories in bundle delivered), pr-manager MUST audit

added a behavioral flip), (c) stale forward-references to prior releases are removed.
This check should block F4→F5 promotion if CHANGELOG is found incomplete.

**Affected files:** pr-manager agent prompt or F4-close checklist template.
**Risk:** Low (additive gate; typical audit is 5-10 lines of diff review).

### PROP-E18-04 — Post-Fixburst Consuming-Surface Sweep as Blocking Checklist (Execution Discipline)

**Category:** agent (implementer/orchestrator fixburst discipline)
**Priority:** P2 HIGH
**Evidence:** PAT-004 recurred (F7-R4 found BC-2.11.028 Architecture Anchors stale after
PR #273 added `#[non_exhaustive]` + `FindingsRender::new`). Policy
DF-CONSISTENCY-AUDIT-POST-FIXBURST-001 was already in the policy registry. The gap is
execution discipline — the fixburst agent does not consult the policy as a blocking
checklist. This is the third recurrence (v0.7.0 x2, v0.9.0 x1).

**Recommendation:** Inject the DF-CONSISTENCY-AUDIT-POST-FIXBURST-001 consuming-surface
sweep as a literal numbered checklist in the fixburst dispatch template (not just as a
policy reference). For any PR that modifies a public API element (struct definition, enum
variants, constructor, `#[non_exhaustive]`, trait impl), the dispatch template must require:

  [ ] BC Architecture Anchors for all BCs governing the changed element — reviewed and updated
  [ ] Story post-delivery notes for all stories whose governing BCs were touched — added
  [ ] VP docs that embed struct construction patterns — reviewed
  [ ] input-hash recomputed for all stories whose `inputs:` BCs were edited

The checklist must be completed and checked off in the PR description before pr-manager
marks the PR ready for adversary re-streak.

**Affected files:** fixburst dispatch template, pr-manager PR checklist template.
**Risk:** Low (codifies existing policy; does not change quality bar, only enforcement).

### PROP-E18-05 — Merge-Authorization Handoff Protocol (D-127 Codification)

**Category:** workflow (orchestrator↔pr-manager)
**Priority:** P2 MEDIUM
**Evidence:** D-127 — pr-manager correctly refused an orchestrator-relayed AUTHORIZE_MERGE
directive (relay-trust guardrail working as designed). Orchestrator merged PR #270 directly.
This is the correct fallback behavior; the issue is the absence of a documented handoff
protocol that makes the correct path explicit rather than discovered at friction time.

**Recommendation:** Document the following merge-authorization handoff protocol in the
feature pipeline sequence:

  1. Human authorizes merge in session (e.g., "merge PR #NNN").
  2. Orchestrator attempts `gh pr merge` directly (not relayed through pr-manager).
     Rationale: pr-manager relay-trust guardrail blocks indirect authorization; direct
     merge by orchestrator under human authorization is the documented clean path.
  3. pr-manager role in F5+ is remediation dispatch (branch creation, test fixes, PR
     creation), not merge authorization relay.
  4. After merge, orchestrator updates STATE.md and notifies pr-manager of new develop HEAD
     via a SendMessage (for logging purposes only, not for authorization).

**Affected files:** STORY-121 scope (E-11 self-improvement), orchestrator feature sequence.
**Risk:** Low (documents existing working behavior; no new tooling required).
