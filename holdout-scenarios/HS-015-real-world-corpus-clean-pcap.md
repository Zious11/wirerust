# Holdout Scenario: Real-World Corpus — Well-Maintained Public PCAP (Low False Positive Rate)

> **WARNING:** This file must NEVER be shown to the implementer or test-writer agents.

## Scenario

1. A forensic analyst runs wirerust on the Wireshark sample capture collection — specifically
   the `http.cap` or `smtp.pcap` files from the Wireshark wiki sample captures page, which
   represent well-formed, legitimate traffic from real-world environments.
2. wirerust processes the file without crashing, panicking, or exiting non-zero.
3. The resulting findings, if any, consist only of genuinely anomalous behaviors present in
   the actual capture — NOT false positives generated by legitimate traffic patterns.
4. The summary output shows a non-zero packet count and byte count matching the capture size.
5. A clean HTTP capture with no anomalies produces zero HTTP findings.

| BC-2.04.003 | Postcondition 1 — FlowKey canonicalization for real bidirectional traffic | Step 4: flow counts are correct |

## Verification Approach

Download from Wireshark sample captures (https://wiki.wireshark.org/SampleCaptures):

```
wirerust analyze --output-format json http.cap | jq '{
  total_packets: .summary.total_packets,
  total_bytes: .summary.total_bytes,
  findings_count: (.findings | length)
}'
```

For `http.cap` (standard HTTP traffic, no anomalies):
- Expect total_packets > 0, total_bytes > 0.
- Expect findings_count == 0 (or very low — e.g., a legitimate large URI should not fire).
- Expect exit code 0.

For `smtp.pcap` (SMTP traffic — not HTTP or TLS, but valid Ethernet capture):
- Expect zero findings (SMTP is not analyzed beyond packet decode).
- Expect exit code 0.

## Evaluation Rubric

- **Functional correctness** (weight: 0.35): Tool processes the real capture without error.
- **Edge case handling** (weight: 0.2): No false positives from legitimate traffic patterns
  (normal HTTP methods, normal URIs, normal Host headers).
- **Data integrity** (weight: 0.3): Packet count and byte count match what a reference tool
  (e.g., tshark -r http.cap -T fields -e frame.len | wc -l) reports.
- **Performance** (weight: 0.15): Tool completes within 5 seconds for a 25KB capture.

## Edge Conditions

- The http.cap may contain HTTP pipelining (multiple requests per TCP connection); these
  should not trigger parse errors.
- If the capture has an HTTP response with a large body, the HTTP analyzer should not
  generate a finding for normal content.

## Failure Guidance

"HOLDOUT LOW: HS-015 (satisfaction: 0.XX) — tool crashes on real-world capture, reports
wrong packet counts, or generates false-positive findings on clean HTTP traffic."
